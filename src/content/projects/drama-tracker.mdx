## Overview

DramaTracker is a full-stack web application for discovering, searching, rating, and tracking movies, powered by the TMDB (The Movie Database) API. Users build personal watchlists, rate films, and browse real-time movie data through a clean, responsive interface. The frontend is built with React, the API with Node.js and Express, data lives in PostgreSQL, and a Python/Flask microservice handles supplemental data processing.

## Problem

Movie enthusiasts often rely on scattered notes, spreadsheets, or memory to track what they have watched and what to watch next. There was no simple, centralized tool that combined live movie data with personal tracking in one clean interface. DramaTracker brings discovery and personal organization together in a single app.

## My Role

I designed and implemented the entire application from scratch as the sole full-stack developer: the React frontend, the Node.js/Express REST API, the PostgreSQL schema, the authentication system, and the Python/Flask data enrichment service.

## Key Features

- **Movie search and discovery.** Real-time search powered by the TMDB API with genre filtering.
- **Watchlist management.** Add, remove, and organize films into custom lists.
- **Rating system.** Rate films individually and view aggregated community scores.
- **Secure authentication.** JWT-based login and session management protecting all user data.

## Tech Stack

- **Frontend:** React with a component-based UI fetching TMDB data through REST calls
- **Backend:** Node.js and Express REST API handling authentication and user data
- **Database:** PostgreSQL schema managing users, watchlists, and ratings
- **Microservice:** Python/Flask service for data enrichment and batch processing
- **External API:** TMDB for live movie metadata

## Development Process

The data model came first: the PostgreSQL schema for users, movies, watchlists, and ratings was designed and revised before significant application code was written. The Express API was then built around clear endpoint contracts, which let the React frontend develop against a stable interface. TMDB integration and server-side caching followed, and the Flask microservice was added last to take batch data enrichment work out of the request path.

## Challenges & Solutions

- **API rate limiting.** Heavy use of the TMDB API risked blowing through its quota. Server-side caching of common queries eliminated redundant calls and noticeably reduced latency at the same time.
- **Database schema design.** The many-to-many relationships between users and movies, with per-user ratings and list membership, took several iterations to model efficiently in PostgreSQL.
- **Full-stack coordination.** Keeping the React frontend and Express backend in sync required disciplined API contracts and testing, so a change on one side could never silently break the other.

## Final Result

A complete, working full-stack application: users sign up, search live movie data, build watchlists, and rate films, all backed by a relational database and a secure authentication flow. The app demonstrates end-to-end ownership of a product across four technology layers.

## What I Learned

- Caching is essential when building on external APIs. Even a simple server-side cache dramatically cut both API usage and response times.
- PostgreSQL was the right choice over NoSQL for this domain. Users, movies, and ratings are inherently relational, and the schema made queries straightforward.
- Planning the data model before writing application logic pays off repeatedly. The iterations spent on the schema saved far more time downstream.
